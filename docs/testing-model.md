# Тестовая модель EduTestAI для ЛР5

## 1. Критические пользовательские сценарии

1. Гость открывает публичную главную страницу и не может попасть в защищенные маршруты без авторизации.
2. Пользователь регистрируется или входит в систему, получает `access_token` и `refresh_token`, а затем восстанавливает сессию через refresh.
3. Пользователь создает тест из текста, получает ссылки Google Forms и видит новую запись в своем списке.
4. Пользователь просматривает, фильтрует, сортирует и листает только свои тесты.
5. Пользователь редактирует и удаляет только свои тесты и вложения.
6. Администратор видит все тесты, всех пользователей и может менять роли.
7. Пользователь загружает вложение допустимого типа, получает ссылку на скачивание и может удалить файл.
8. Внешние интеграции `OpenRouter`, `Google Forms` и S3 работают штатно либо возвращают контролируемую ошибку без падения приложения.

## 2. Ключевые бизнес-правила и ограничения

- Доступ к защищенным API разрешен только по Bearer-токену и активному пользователю.
- Роль `user` работает только со своими тестами и вложениями.
- Роль `admin` может просматривать все тесты и управлять ролями пользователей.
- Первый зарегистрированный пользователь автоматически становится `admin`.
- Refresh-токен хранится в `HttpOnly` cookie, а при обновлении сессии происходит ротация refresh-токена.
- При `logout` текущая refresh-сессия отзывается, повторный refresh запрещен.
- Генерация теста невозможна для пустого текста и для набора вопросов меньше трех.
- Вложения принимаются только для разрешенных расширений и MIME-типов, размер ограничен `10 МБ`.
- Для списка тестов обязательна корректная работа фильтрации, сортировки и пагинации.

## 3. Области повышенного риска

- Аутентификация: ошибки логина, истечение access-токена, refresh без cookie, повторный refresh после logout.
- RBAC: доступ пользователя к чужим тестам, доступ не-админа к `/api/users` и `/app/admin`.
- Работа с файлами: запрещенные типы, превышение размера, удаление файлов из S3 и очистка записей в БД.
- Внешние API: отказ `OpenRouter`, отказ `Google Forms`, сбои объектного хранилища.
- UI-состояния: лоадеры, ошибки загрузки, очистка состояния после протухшей сессии, скрытие admin-элементов.

## 4. Стратегия тестирования

### Backend

- `unit`: сервисный слой и чистая бизнес-логика без HTTP.
- `integration`: FastAPI endpoint'ы с тестовой БД и моками внешних зависимостей.

Приоритетные проверки:

- bootstrap первого администратора;
- ротация refresh-токенов;
- валидация файлов;
- коды ответа `200/201/204/400/401/403/404/422/503`;
- ограничения "только свои данные";
- поведение внешних интеграций при отказах.

### Frontend

- `unit`: API-утилиты, маршрутизация, роль-зависимое отображение.
- `integration`: сценарии компонентов со взаимодействием пользователя.

Приоритетные проверки:

- защита маршрутов `ProtectedRoute` и `AdminRoute`;
- показ admin-меню только для администратора;
- синхронизация фильтров страницы тестов с URL;
- повтор запроса после `401` и refresh;
- очистка локального auth-state при неуспешном refresh.

### E2E

- вход в систему и переход в кабинет;
- открытие списка тестов и применение фильтров;
- переход администратора к управлению пользователями;
- восстановление или сброс сессии на уровне UI при ответах API.

## 5. Тестовая инфраструктура

- Backend использует отдельные тестовые зависимости и изолированную SQLite БД через `pytest`.
- Frontend использует `Vitest` + `Testing Library` для быстрых проверок.
- Сквозные проверки выполняются через `Playwright` с мокированием сетевых ответов.
- Внешние зависимости `OpenRouter`, `Google Forms` и S3 изолируются через `monkeypatch` или browser route mocks.
- Состояние между тестами очищается фикстурами БД, `localStorage`, cookie и reset моков.

## 6. Правила структуры и метрики

- Backend-тесты: `backend/tests/unit` и `backend/tests/integration`.
- Frontend-тесты: рядом с кодом в `src/**/*.test.ts(x)`.
- E2E-тесты: `frontend/e2e`.
- Имена тестов описывают бизнес-ожидание, а не внутреннюю реализацию.
- Минимальные целевые команды покрытия:
  - backend: `pytest --cov=app --cov-report=term-missing`;
  - frontend: `vitest run --coverage`.

## 7. Критерии итоговой проверки

- Ключевые пользовательские сценарии проходят без регрессий.
- Доступ к чужим данным и admin-функциям для обычного пользователя заблокирован.
- Ошибки интеграций и сессии обрабатываются предсказуемо и не ломают интерфейс.
